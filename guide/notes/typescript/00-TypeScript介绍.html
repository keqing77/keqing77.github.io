# TypeScript

TypeScript其实就是给JavaScript加上了类型系统

TS其实用到的大多数内容只有列出的这些

这里只介绍TS独有的，很多概念实际在ES6或者后面的标准都有且通用的 如 面向对象的 类class 、装饰器decorator等等

# TypeScript是JavaScript的超集

  • 这意味着你完全可以在TypeScript中写JavaScript，完全兼容JS代码
  • TypeScript需要编译转化成JavaScript代码，浏览器不识别TS代码
  • 刚开始的时候都是先用JS写好，再在上面添加约束

# TypeScript的类型系统

  • 类型系统按照「是否允许隐式类型转换」来分类，可以分为强类型和弱类型
  • JS是弱类型的脚本语言，由于隐式类型转换的存在，我们很难在运行前确定变量的类型
  • TS中一旦定义了类型，就确定每个变量的类型，不能随意改变，使得在维护大型项目时候更加直观。
  • TS仍然是弱类型的，依然保留JS运行时的特性

# TypeScript是静态类型

JS代码有错需要运行才知道，而TS在写代码的时候就会检查错误并有错误提示反馈给你

JS运行时编译，也就是动态编译

const js = 'goodLanguage'
js.splice('')  
// js是字符串类型，没有splice方法，但你只有在运行时候才会报错

# TypeScript与标准同步发展

同步标准

  • TypeScript变得这么流行的原因是坚持与 ECMAScript 标准同步发展。
  • ECMAScript 是 JavaScript 核心语法的标准，自 2015 年起，每年都会发布一个新版本，包含一些新的语法。
  • 以闭源垄断著称的微软 从不遵守标准的IE 变成 如今紧跟行业标准在开源界混得风生水起🤣

一个新的语法从提案到变成正式标准，需要经历以下几个阶段：

  • Stage 0：展示阶段，仅仅是提出了讨论、想法，尚未正式提案。
  • Stage 1：征求意见阶段，提供抽象的 API 描述，讨论可行性，关键算法等。
  • Stage 2：草案阶段，使用正式的规范语言精确描述其语法和语义。
  • Stage 3：候选人阶段，语法的设计工作已完成，需要浏览器、Node.js 等环境支持，搜集用户的反馈。
  • Stage 4：定案阶段，已准备好将其添加到正式的 ECMAScript 标准中。
